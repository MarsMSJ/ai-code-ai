# AV1 Decoding Process - Summary

## Overview

AV1 contains two operating modes:
1. **General decoding**: Input is a sequence of OBUs, output is decoded frames
2. **Large scale tile decoding**: Input is a tile list OBU plus side information, output is a decoded frame

### General Decoding Process

When `film_grain_params_present` is 0, decoders produce output frames identical in all respects with same output order.

When `film_grain_params_present` is 1, decoders implement film grain synthesis modifying OutY, OutU, OutV arrays. Two conformance options:
- Exact implementation matching reference process
- Intermediate frames matching reference, with perceptually equivalent final output

**Input**: Sequence of OBUs  
**Output**: Sequence of decoded frames

For each OBU, syntax elements are extracted per section 5.3. Syntax tables include function calls indicating when decode processes are triggered.

## Large Scale Tile Decoding Process

Decodes random subset of tiles from multiple coded frames. Tile list specified by tile list OBU.

**Inputs**:
- Syntax elements from sequence header OBU
- Syntax elements from frame header OBU (including CDF tables)
- AnchorFrames array (up to 128 frames)
- Tile list OBU

**Output**: Output frame containing decoded tiles in raster order

**Process**: For each tile list entry:
1. Parse tile_list_entry syntax elements
2. Set bitstream position to coded_tile_data start
3. Initialize reference frame variables from anchor frame
4. Invoke decode camera tile process
5. Write decoded tiles to output frame in raster order

**Conformance requirements**: Many features disabled (superres, order hint, film grain, restoration, CDEF, etc.). Tiles must be exactly one superblock high, uniform width.

### Decode Camera Tile Process

Decodes single tile within frame. Outputs OutY, OutU, OutV arrays.

Process: Initialize QIndex, symbol decoder, iterate superblocks in raster order, decode partitions. Does not invoke post-processing (deblock, CDEF, superres, loop restoration, reference frame update).

## Decode Frame Wrapup Process

Triggered by `decode_frame_wrapup` call. Performs frame-level decode after all tile-level decode.

If `show_existing_frame == 0`, ordered post-processing steps:
1. Loop filter (if levels non-zero)
2. CDEF process (CurrFrame → CdefFrame)
3. Upscaling (CdefFrame → UpscaledCdefFrame)
4. Upscaling (CurrFrame → UpscaledCurrFrame)
5. Loop restoration (UpscaledCurrFrame + UpscaledCdefFrame → LrFrame)
6. Motion field motion vector storage
7. Segmentation map update (if needed)

Then:
1. Reference frame update process
2. Output process (if show_frame or show_existing_frame)

## Ordering of OBUs

OBUs must appear in specific order:
- Sequence header OBU (optional, but required for random access)
- Frame header OBU
- Metadata OBUs (optional)
- Tile group OBUs
- Frame OBU (optional)
- Tile list OBU (optional)

Sequence headers may appear multiple times within coded video sequence (must be bit-identical except operating_parameters_info).

## Random Access Decoding

**Key frame random access point**: frame_type=KEY_FRAME, show_frame=1, contains sequence header OBU

**Delayed random access point**: frame_type=KEY_FRAME, show_frame=0, contains sequence header OBU

**Key frame dependent recovery point**: show_existing_frame=1, references delayed random access point

**Conformance**: Decoders must support starting at key frame or delayed random access points.

## Frame End Update CDF Process

Triggered by `frame_end_update_cdf`. Copies saved CDF arrays back to frame CDF arrays (for CDF arrays in init_coeff_cdfs and init_non_coeff_cdfs).

## Set Frame Refs Process

Triggered by `set_frame_refs`. Computes ref_frame_idx array based on:
- last_frame_idx, gold_frame_idx (explicit)
- RefOrderHint array (output order hints)

Process:
1. Initialize ref_frame_idx with LAST_FRAME and GOLDEN_FRAME from explicit indices
2. Compute shiftedOrderHints (relative to current frame)
3. Set ALTREF_FRAME to latest backward reference
4. Set BWDREF_FRAME to earliest backward reference
5. Set ALTREF2_FRAME to next closest backward reference
6. Set remaining forward references in anti-chronological order
7. Fill any remaining with smallest output order

## Motion Field Estimation Process

### General

Estimates motion field for current frame using reference frames. Used for motion vector prediction.

### Projection Process

Projects motion vectors from reference frames to current frame using global motion parameters. Handles:
- Translation
- Rotation/zoom
- Affine transforms

### Get MV Projection Process

Computes projected motion vector for a block position in reference frame to current frame coordinates.

### Get Block Position Process

Determines block position in reference frame given current block position and motion vector.

## Motion Vector Prediction Processes

### Find MV Stack Process

Builds motion vector candidate stack for current block. Searches:
- Spatial neighbors (above, left, above-left, above-right, below-left)
- Temporal samples from reference frames
- Global motion vectors
- Compound motion vectors
- Extra search candidates

Process includes:
- Setup global MV
- Scan row/col for spatial candidates
- Scan point for temporal candidates
- Add reference motion vectors
- Search stack (remove duplicates, clamp)
- Compound search stack
- Lower precision conversion
- Sorting by priority
- Extra search for additional candidates
- Context and clamping

### Has Overlappable Candidates Process

Determines if motion vector candidates allow overlapped motion compensation.

### Find Warp Samples Process

Finds samples for warp estimation. Collects samples from reference frames using motion vectors.

## Prediction Processes

### Intra Prediction Process

Predicts samples using already reconstructed samples in current frame.

**Modes**:
- **Directional**: 8 directions (angles)
- **DC**: Average of left and above samples
- **Smooth**: Smooth interpolation
- **Smooth V/H**: Vertical/horizontal smooth
- **Recursive**: Filter-based prediction (filter_intra)
- **Paeth**: Basic prediction using left, above, above-left

**Process**:
1. Derive AboveRow and LeftCol arrays from neighbors
2. Apply prediction mode
3. Optional edge filtering (intra edge filter)
4. Write predicted samples to CurrFrame

**Intra Edge Filter**: Optional filtering of prediction edges using strength selection and upsample processes.

### Inter Prediction Process

Predicts samples using reference frames and motion vectors.

**Types**:
- **Block inter prediction**: Standard motion compensation
- **Block warp**: Warped motion compensation (affine transforms)
- **Overlapped motion compensation**: Blending multiple predictions

**Process**:
1. Derive rounding variables
2. Scale motion vectors (if needed)
3. Perform block inter prediction or warp
4. Optional overlapped motion compensation with blending

**Warp Estimation**: Estimates warp parameters from motion vectors.

**Overlap Blending**: Blends predictions using masks:
- Wedge mask
- Difference weight mask
- Intra mode variant mask
- Distance weights

### Palette Prediction Process

Predicts samples using color palette. Decodes palette colors and indices.

### Predict Chroma from Luma Process

Predicts chroma samples from reconstructed luma samples using linear model.

## Reconstruction and Dequantization

### General

Reconstructs samples from quantized transform coefficients.

### Dequantization Functions

- `get_dc_quant(plane)`: Returns DC quantization step
- `get_ac_quant(plane)`: Returns AC quantization step

### Reconstruct Process

1. Derive quantization parameters (with optional quantization matrix)
2. Dequantize coefficients: `dq = Quant[i][j] * q2 / dqDenom`
3. Apply 2D inverse transform
4. Add residual to prediction: `CurrFrame = Clip1(Prediction + Residual)`

Handles transform size-dependent dqDenom (1, 2, or 4).

## Inverse Transform Process

### 1D Transforms

**Butterfly Functions**: B() performs rotation using cos128/sin128 lookup tables. H() performs Hadamard rotation.

**Inverse DCT**: Uses butterfly functions with bit-reversal permutation. Supports sizes 4, 8, 16, 32, 64.

**Inverse ADST**: Asymmetric discrete sine transform. Includes input/output array permutations. Supports sizes 4, 8, 16.

**Inverse Walsh-Hadamard**: Uses H() function.

**Inverse Identity**: Pass-through transform (no operation).

### 2D Inverse Transform Process

Applies 1D transforms horizontally then vertically based on PlaneTxType:
- DCT_DCT, DCT_ADST, ADST_DCT, ADST_ADST
- FLIPADST variants
- H_FLIPADST, V_FLIPADST
- IDTX variants

## Loop Filter Process

### General

Deblocking filter applied to block edges. Operates on CurrFrame.

### Edge Loop Filter Process

Filters vertical edges then horizontal edges. Processes:
- Luma edges (4x4, 8x8, 16x16 block boundaries)
- Chroma edges (2x2, 4x4, 8x8 block boundaries)

### Filter Size Process

Determines filter size (4, 8, or 16) based on block size and edge type.

### Adaptive Filter Strength Process

Computes filter strength based on:
- Loop filter levels
- Reference frame deltas
- Mode deltas
- Segmentation (if enabled)

### Sample Filtering Process

Applies filtering using:
- Filter mask process (determines which samples to filter)
- Narrow filter (4-tap)
- Wide filter (7-tap)

Filtering uses threshold-based decisions on edge strength.

## CDEF Process

Constrain Directional Enhancement Filter. Applied after loop filter.

### CDEF Block Process

Processes blocks using directional filtering. Determines:
- Filter direction
- Filter strength
- Filter coefficients

### CDEF Direction Process

Finds dominant direction for filtering using gradient analysis.

### CDEF Filter Process

Applies directional filter along detected direction. Uses separable filtering.

## Upscaling Process

Upscales frames using super-resolution. Applied when `enable_superres == 1`.

Process:
1. Determine upscaling ratio
2. Apply upscaling filter
3. Produce UpscaledWidth x FrameHeight output

## Loop Restoration Process

Final restoration filter. Applied after upscaling.

### Loop Restore Block Process

Processes restoration units. Selects filter type:
- Self guided filter
- Box filter
- Wiener filter

### Self Guided Filter Process

Uses guided filtering with local statistics.

### Box Filter Process

Simple box/average filtering.

### Wiener Filter Process

Applies Wiener filter with learned coefficients. Uses:
- Wiener coefficient process (decodes coefficients)
- Get source sample process (retrieves samples)

## Output Process

### General

Prepares final output frames. Handles:
- Intermediate output preparation
- Film grain synthesis (if enabled)

### Intermediate Output Preparation Process

Prepares OutY, OutU, OutV arrays from LrFrame. Handles:
- Frame dimensions
- Subsampling (chroma)
- Bit depth conversion

### Film Grain Synthesis Process

Synthesizes film grain noise and adds to output.

**Process**:
1. **Random number process**: Generates random numbers using AR model
2. **Generate grain process**: Generates grain patterns using:
   - Scaling lookup initialization
   - Noise stripe generation
   - Stripe blending
3. **Add noise synthesis process**: Blends grain with image:
   - Intensity-dependent scaling
   - Chroma scaling from luma (optional)
   - Clipping to valid range

Uses piecewise linear scaling lookup tables. Grain strength varies with intensity.

## Motion Field Motion Vector Storage Process

Filters and reorders motion vectors for storage. Stores:
- MfRefFrames: Reference frame indices
- MfMvs: Motion vectors

Only odd row/col positions affect decoding.

## Reference Frame Update Process

Saves current frame state to reference frames. Triggered for frames where `refresh_frame_flags` bit is set.

**Stored**:
- Frame dimensions (width, height, upscaled width, render size)
- MI dimensions (MiCols, MiRows)
- Subsampling, bit depth
- Frame type, order hint
- FrameStore (decoded samples)
- SavedRefFrames, SavedMvs (motion vectors)
- SavedGmParams (global motion parameters)
- SavedSegmentIds (segmentation map)
- CDF arrays (save_cdfs)
- Film grain parameters (if enabled)
- Loop filter parameters
- Segmentation parameters

## Reference Frame Loading Process

Loads saved reference frame state into current frame variables. Reverse of update process.

Used when `show_existing_frame == 1` and `frame_type == KEY_FRAME`.

Loads all state variables from FrameStore indexed by `frame_to_show_map_idx`.
